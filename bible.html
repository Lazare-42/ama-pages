1. Introduction

Ce document est la référence centrale du projet AMA.
Il décrit l'intégralité de l'arborescence, les fichiers clés, les workflows, et permet à n'importe qui (humain ou IA) de comprendre et continuer le travail.

Pourquoi ce document ?

  • Migration de workspace — Tout est documenté pour changer de machine
  • Continuité Claude — Une nouvelle session peut reprendre immédiatement
  • Collaboration — Un tiers peut comprendre le projet sans explication orale
  • Mémoire — Éviter de refaire ce qui a déjà été fait

11. Processus de Mise à Jour

Règle de cohérence

Une info mise à jour dans 1 fichier = mise à jour dans TOUS les fichiers concernés.

Exemple : si un statut passe de "proto" à "testé", modifier les 4 portfolios + STRATEGIE + ROADMAP.

Checklist mise à jour

Nouvelle formation / expérience :
  → PROFIL-YANN-REFERENCE.md
  → STRATEGIE-2026-2027.md (si impact écoles)
  → 4 portfolios HTML
  → AMA-PROJECT-BIBLE.html

Nouveau projet AMA :
  → PROFIL-YANN-REFERENCE.md (mapping thèmes)
  → BANQUE-ARGUMENTS-THEMATIQUES.md
  → AMA-ROADMAP.md
  → 4 portfolios HTML

Changement de statut projet :
  → 4 portfolios HTML
  → STRATEGIE-2026-2027.md
  → AMA-ROADMAP.md

Après un test / synthèse :
  → BANQUE-ARGUMENTS-THEMATIQUES.md (nouveaux arguments)
  → STRATEGIE-2026-2027.md (leçons apprises)

12. Guide de Migration & Disaster Recovery

Ce guide couvre 4 scénarios : migration machine, migration agent (personne), migration structure, et disaster recovery. Voir aussi Docs/INVENTAIRE.md pour la liste complète des fichiers.

12.A — Migration Machine

La clé USB fonctionne, on change juste d’ordinateur.

Étape 1 — Créer un backup frais

cd AMA && bash backup.sh

L’archive datée est créée dans ~/AMA-backups/.

Étape 2 — Brancher la clé sur la nouvelle machine

# La clé se monte généralement dans /run/media/ ou /media/
ls /run/media/$USER/KINGSTON/code/yann/AMA/AMA-PROJECT-BIBLE.html

Étape 3 — Vérifier l’état

cd AMA && bash start.sh

Le script vérifie tous les fichiers clés, repos git, outils installés.

Étape 4 — Installer les dépendances manquantes

# Selon le résultat de start.sh :
# - Go (pour INDEX backend)
# - Node.js (pour index-dashboard)
# - PostgreSQL (pour index-messaging)
# - signal-cli (pour le pont Signal)
# - curl (pour uploads 0x0.st)

Étape 5 — Recréer les .env

Les fichiers .env ne sont pas versionnés. Recréer :

# INDEX/index-messaging/.env
PORT=8080
HOST=localhost
# + credentials PostgreSQL, Signal config

12.C — Migration Structure (AMA absorbé par une autre entité)

AMA est intégré dans une organisation, un studio, ou un projet plus large.

Ce qui est portable

  • INDEX/ — 6 repos git indépendants, déplaçables tels quels
  • Docs/ — Markdown standard, lisible partout
  • Portfolios/ — HTML autonomes, pas de dépendances externes
  • Les scripts — backup.sh et start.sh utilisent des chemins relatifs

Ce qui doit être adapté

  • .env et credentials — À reconfigurer pour le nouvel environnement
  • Liens 0x0.st — Relancer bash backup.sh --upload pour générer de nouveaux liens
  • Références à « KINGSTON » — Les scripts utilisent des chemins relatifs, mais certains docs référencent la clé
  • GitHub remotes — Mettre à jour si le repo change d’organisation

Procédure

1. Copier AMA/ vers la nouvelle structure
2. bash start.sh                        # Vérifier l'état
3. Mettre à jour les GitHub remotes si besoin :
   cd INDEX/index-messaging && git remote set-url origin NEW_URL
4. Recréer les .env pour le nouvel environnement
5. bash backup.sh --upload              # Nouveaux liens 0x0.st
6. Mettre à jour cette Bible (section 2, contacts, timeline)

12.D — Disaster Recovery (la clé USB meurt)

La clé KINGSTON est perdue, cassée ou corrompue. Que reste-t-il ?

Ce qui survit

QuoiOùComplet ?
Code INDEX (6 repos)GitHubOui (sauf branches locales non pushées)
HTML publics0x0.stTemporaire — liens expirent
Dernier backup~/AMA-backups/Si backup.sh a été lancé récemment

Ce qui est perdu (sans backup)

  • 44+ fichiers Docs/ (stratégie, profil, arguments, contacts...)
  • 4 portfolios HTML + 2 landing pages + questionnaire + dossier KLIMA
  • Bible AMA, Dashboard, Conciergerie
  • Collabs, Music, METR, JOE (~1.5 Go de médias)
  • Tous les .env et configurations locales

Procédure de relance complète

# 1. Préparer le nouveau support
mkdir -p /chemin/nouveau/AMA

# 2. Récupérer depuis le backup (si disponible)
cd /chemin/nouveau
tar xzf ~/AMA-backups/AMA-backup-DERNIER.tar.gz

# 3. Sinon, reconstruire depuis GitHub
mkdir -p AMA/INDEX && cd AMA/INDEX
git clone git@github.com:USER/index-messaging.git
git clone git@github.com:USER/index-dashboard.git
git clone git@github.com:USER/index-resources.git
git clone git@github.com:USER/index-shared.git
git clone git@github.com:USER/index-tasks.git
git clone git@github.com:USER/index-users.git

# 4. Vérifier l'état
cd /chemin/nouveau/AMA && bash start.sh

# 5. Recréer les fichiers manquants
# - .env (voir section 7 de INVENTAIRE.md)
# - BRIEFING-QUOTIDIEN.md (réécrire)
# - Les Docs/ devront être recréés si pas de backup

# 6. Relancer les liens publics
bash backup.sh --upload

12.E — Prévention

Pour éviter la perte de données :

  • Lancer bash backup.sh au moins 1x/semaine
  • Configurer BACKUP_DEST vers un dossier hors de la clé USB (disque dur, cloud mount)
  • Vérifier les liens 0x0.st régulièrement (bash backup.sh --upload)
  • Pusher les branches INDEX avant de débrancher la clé
  • Consulter Docs/INVENTAIRE.md pour la liste complète de ce qui n’est pas protégé
